# Heap & Stack

## 메모리 구조
- RAM
    - Code
        - 소스 코드가 기계어로 저장되는 곳, 컴파일 단계에서 결정되며 코드가 수정되지 않도록 Read-Only 형태로 저장됨
    - Data
        - 전역변수, static 변수가 저장되는 곳, 프로그램 시작과 동시에 할당되며 프로그램이 종료될때 해제됨, 값이 변경될 수 있어 Read-Write 형태로 저장됨
    - Heap
        - Class, Closure 등 크기가 크고 여러곳에서 **참조 타입**이 저장되는 곳
        - 런타임 시에 할당된다.
        - 사람이 직접 할당, 해제 할 수 있다 (동적 할당).
        - **사용 후 반드시 메모리 해제를 해줘야 한다.**
        - Swift에서는 **ARC(Automatic Reference Counting)**을 통해 참조되지 않으면 자동으로 Heap에서 해제를 해준다.
        
        - 장점
            - 메모리 크기에 **제한 없음**
            - 프로그램 내부의 모든 함수에서 참조 가능
            
        - 단점
            - 메모리 할당, 해제로 인한 상대적 속도 느림
            - 경합 (Lock) 으로 인한 상대적 속도 느림
            - 메모리를 **직접 관리** 해야함
                - 관리 해주지 않으면 **Memory Leak 발생**
    - Stack
        - 지역 변수, 함수의 매개변수, 리턴값, Struct 등이 저장되는 곳
        - 컴파일 단계에서 결정되며 **무한히 할당할 수 없다**.
        - 함수( `}` )가 종료되면 **자동으로 메모리에서 해제** 된다.
        - Last In First Out 의 데이터 구조를 가지고있다.
        - CPU에서 관리, 최적화 된다.
        
        - 장점
            - CPU가 Stack 메모리를 효율적으로 구성하기 때문데 상대적 속도 빠름
            - 메모리를 **직접 해제 해주지 않아도 됨**
            
        - 단점
            - 메모리의 크기에 제한이 있음
            - 지역 변수만 메모리에 저장 가능
            
## Heap VS Stack

### Heap, Stack 사용
- Heap은 언제 사용해야 하나
    - 데이터의 크기를 모를때 사용
    - 스택에 저장하기에는 큰 데이터의 경우에 Heap에 할당
        - **Instance, Closer 등 자동으로 할당되는것 외에 직접 할당할 경우**
        - Heap에 너무 많은 메모리가 할당될 경우 **Heap Over Flow** 발생
        
- Stack은 언제 사용해야 하나
    - Heap 에 할당하는 상황을 제외 하고는 Stack에 할당하면 됨
        - Stack에 너무 많은 메모리가 할당될 경우 **Stack Over Flow** 발생

### 관계
- Heap과 Stack은 결국 같은 메모리 영역을 공유함
    - Heap은 낮은 메모리 주소(`0x00000000`) 부터 할당
    - Stack은 높은 메모리 주소(`0xffffffff`) 부터 할당
